Pi 5 Hardware Monitor — BUILD-NOTES

This file is for source/build and project-maintenance reference.
It is not needed by normal users who are only installing the plugin.

Project identity

* Public package/install name: pi-monitor
* Development history used the v2 name internally, but public install/package naming should stay pi-monitor

Public install flow

* Normal users should install from the repo root with:
  sudo ./install.sh
* Do not direct normal users to make devel-install
* The installer uses the starter-kit Makefile install flow plus the history collector/service/timer install

Current installed/runtime paths

* Cockpit plugin:
  /usr/local/share/cockpit/pi-monitor
* History collector script:
  /usr/local/bin/pi-monitor-history
* Service unit:
  /etc/systemd/system/pi-monitor-history.service
* Timer unit:
  /etc/systemd/system/pi-monitor-history.timer
* Runtime history data:
  /var/lib/pi-monitor/history.ndjson

Current installer behavior

* verify it is run with sudo/root
* verify it is run from the repo root
* hard-stop unless the system is a Raspberry Pi 5
* verify Cockpit is present via cockpit-bridge
* verify or install required build/runtime dependencies as needed
* optionally offer smartmontools for full NVMe SMART details
* run make install
* install the history collector and unit files from the repo copies
* set the collector script executable
* run systemctl daemon-reload
* run systemctl enable --now pi-monitor-history.timer
* run systemctl start pi-monitor-history.service once so fresh installs get an immediate first history sample
* do not restart Cockpit automatically

Required items currently enforced by install.sh

* Raspberry Pi 5 hardware
* cockpit-bridge
* systemctl
* make
* nodejs
* npm
* python3
* vcgencmd

Optional item currently offered by install.sh

* smartmontools

  * purpose: full NVMe SMART health details such as health status, temperature, percentage used, power-on hours, unsafe shutdowns, and media/data integrity errors

Why the immediate first service start matters

* Without it, a fresh install can look broken because History / Trends stays blank until the timer's first scheduled run.

Fresh-install verification flow

Negative hardware test:

* Use a non-Pi-5 system and confirm:
  sudo ./install.sh
* Expected result:
  installer exits cleanly with the non-Pi-5 hardware message and does not continue

Positive Pi 5 install test from a clean clone:
cd /home/asl
sudo rm -rf cockpit-pi5-monitor
git clone https://github.com/ke2hni/cockpit-pi5-monitor.git
cd cockpit-pi5-monitor
sudo ./install.sh

Then verify:
cockpit-bridge --packages | grep -A3 -B2 'pi-monitor|Pi 5 Hardware Monitor'
systemctl status pi-monitor-history.timer --no-pager
systemctl status pi-monitor-history.service --no-pager
ls -ld /var/lib/pi-monitor /var/lib/pi-monitor/history.ndjson
sudo tail -n 3 /var/lib/pi-monitor/history.ndjson

Expected results

* plugin appears as pi-monitor in cockpit-bridge --packages
* install path is /usr/local/share/cockpit/pi-monitor
* timer is enabled and active
* service exits successfully as a oneshot
* history.ndjson exists and contains samples immediately after install
